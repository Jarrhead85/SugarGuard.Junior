# SugarGuard — Описание проекта v3.2

> Актуально на 21.07.2026. Документ описывает текущую реализацию SugarGuard.Junior: API, Web-кабинеты родителя/врача/администратора, детское мобильное приложение, Telegram- и MAX-боты, поддержку, интеграцию Android с Juggluco и production-деплой.

## Общее описание

SugarGuard — система мониторинга и ведения журнала диабета у детей. Включает детское мобильное приложение (.NET MAUI), веб-кабинеты родителя, врача и администратора (Blazor Server), REST API (ASP.NET Core), Telegram-бот и интеграцию с ботом MAX для уведомлений.

Система поддерживает ручной ввод показателей глюкозы и приём потока CGM через Juggluco на Android, дневник питания и инсулина, расписание приёмов пищи и перекусов, управление рюкзаком с ХЕ, привязку родителей и врачей через коды, AI-рекомендации GigaChat, контроль диалогов ребёнка с ИИ родителем, Telegram/MAX/email/web-уведомления, ачивки для ребёнка, поддержку, статьи, аудит действий и экспорт данных в CSV/PDF.

SugarGuard не является медицинским изделием и не заменяет врача. Интеграция Libre предназначена для журнала и информирования: приложение не активирует сенсор, не управляет инсулином и не выполняет самостоятельных медицинских расчётов.

## Технологический стек

### API (SugarGuard.API)

- **Фреймворк**: ASP.NET Core Web API, .NET 9
- **БД**: PostgreSQL (прод) + SQLite (разработка), EF Core 9 с миграциями
- **Аутентификация**: JWT Bearer + Refresh Token Rotation
- **Шифрование**: AES-256-GCM (PHI-данные: имя, email, фамилия пользователя); HMAC-SHA256 (коды подключения); PBKDF2-SHA256 600 000 итераций (пароли)
- **Фоновые задачи**: Hangfire + PostgreSQL (прод), inline-fallback (SQLite/dev)
- **AI**: GigaChat API (Sber) с OAuth2, retry-политикой Polly и circuit breaker; local-fallback при недоступности
- **Email**: SMTP через `IEmailService` (настроен на smtp.yandex.ru:465 SSL)
- **Уведомления**: Telegram Bot API, MAX Bot API, email, web-уведомления в кабинете и Web Push (VAPID)
- **Rate Limiting**: `FixedWindowLimiter` — `auth-login` (5 req/min/IP, 429 с Retry-After), `bot-login` отдельный лимит
- **Документация**: Swagger / OpenAPI v1 с JWT Bearer security scheme
- **Логирование**: `ILogger<T>`, HttpLogging middleware (метод + путь + статус + длительность)

### MAUI (SugarGuard.Junior)

- **Архитектура**: MVVM, CommunityToolkit.Mvvm
- **HTTP**: `IApiClient` / `RealApiClient`, `IHttpClientFactory`, `JwtDelegatingHandler`
- **Хранилище**: `SecureStorage` (токены и служебные состояния), SQLite (зашифрованные локальные данные и офлайн-очередь синхронизации)
- **CGM на Android**: manifest `BroadcastReceiver` принимает документированные broadcast Juggluco `glucodata.Minute` и xDrip-совместимый `com.eveningoutpost.dexdrip.BgEstimate`; чтение закрытого протокола Libre не реализуется
- **Графики**: `Microcharts.Maui`, `GlucoseChartDrawable` (кастомный `IDrawable`)
- **Сериализация**: `System.Text.Json`
- **UI**: `AppThemeBinding`, `Colors.xaml`, `Typography.xaml`, детские hero-карточки, маскот, скины интерфейса; шрифты Nunito + Montserrat в MAUI

### Blazor (SugarGuard.Web)

- **Фреймворк**: Blazor Server, .NET 9
- **Аутентификация**: `JwtAuthStateProvider`, `JwtAuthorizationHandler`, `LocalStorageTokenStore`
- **HTTP**: `IHttpClientFactory`, `SugarGuardApiService`
- **Локализация**: `AddLocalization`, `Strings.ru.resx`
- **UI**: единый shell для Parent/Doctor/Admin, синхронизированные боковая и верхняя панели навигации, карточки KPI, адаптивные sidebar/topbar, Chart.js, кастомные компоненты `StateBanner`, `EmptyStateView`, `SyncBanner`

## Реализованные модули

### 1. Аутентификация и безопасность

- **Регистрация**: email + пароль, `IsEmailVerified` флаг; PHI-поля (имя, фамилия, email) шифруются AES-256-GCM через `ICryptoService` перед сохранением в БД
- **Вход**: `POST /api/auth/login` — проверка PBKDF2, возврат access-токена + refresh-токена; rate-limit 5 req/min/IP
- **Refresh Token Rotation**: `POST /api/auth/refresh` — при ротации старый токен отзывается, все токены пользователя отзываются при компрометации или деактивации аккаунта
- **Выход**: `POST /api/auth/logout` — отзыв конкретного refresh-токена
- **Сброс пароля и подтверждение email**: 8-символьный код формата `ABCD-1234`, TTL 15 мин, `CryptographicOperations.FixedTimeEquals` для защиты от timing-атак, `VerificationService` на `IMemoryCache` (single-instance, caveat задокументирован)
- **Bot-login**: `POST /api/auth/bot-login` — API-ключ для Telegram-бота, создаёт/получает `ServiceAccount`
- **RBAC**: политики `ParentOrDoctorOrAdmin`, `AdminOnly`, `DoctorOrAdmin` на уровне контроллеров

### 2. Показатели глюкозы (Measurements)

- CRUD: `POST/GET/PUT/DELETE /api/measurements`
- Фильтрация по дате, `uiState` (Normal / Warning / Critical), пагинация
- Статусы: `CriticallyLow`, `Low`, `Normal`, `High`, `CriticallyHigh` — через `GlucoseStatusService` и `GlucoseUiStateService` (общие константы в `SugarGuard.Shared`)
- Для ручного измерения ребёнка может быть запрошена AI-рекомендация: `GigaChatService` → `LocalFallbackService`; потоковые показания CGM не создают рекомендацию на каждую минуту
- При критическом измерении приложение пытается получить координаты с разрешения пользователя; сервер сохраняет уведомление в кабинете родителя и отправляет его в подключённые каналы Telegram/MAX
- Поле `NotifyParents` позволяет сохранять измерение без немедленного оповещения; это необходимо для непрерывного потока CGM
- API не отправляет повторное уведомление при идемпотентной повторной синхронизации измерения с тем же `MeasurementId`

#### Поток CGM: FreeStyle Libre через Juggluco на Android

- Поддерживаемый программный путь — локальный Android broadcast, который публикует Juggluco после получения показания сенсора: `glucodata.Minute` либо `com.eveningoutpost.dexdrip.BgEstimate`
- `JugglucoGlucoseBroadcastReceiver` принимает broadcast вне UI-потока; `JugglucoBroadcastReadingParser` приводит глюкозу к ммоль/л, нормализует время UTC, проверяет диапазон и отвергает сообщения с некорректным временем из будущего
- Значение сначала сохраняется в зашифрованную локальную SQLite-базу как `DataSource.CGMSystem`, затем добавляется в стандартную офлайн-очередь `SyncQueueItem`; сеть не требуется для сохранения
- Повтор одного показания исключается по ребёнку, источнику `CGMSystem` и времени измерения в окне ±1 секунды
- Для непрерывной критической ситуации родительское оповещение создаётся при входе в критическое состояние и затем не чаще одного раза в 15 минут; нормальное показание сбрасывает локальный маркер критического состояния
- SugarGuard не подключается напрямую к NFC/Bluetooth-протоколу Libre, не извлекает закрытые библиотеки Juggluco, не активирует сенсор и не заявляет подтверждённую совместимость с конкретной моделью Libre до полевого тестирования
- Для настройки на телефоне пользователь устанавливает Juggluco, настраивает в нём совместимый сенсор и выбирает SugarGuard в `Glucodata broadcast` или `xDrip broadcast`. Для Android 11+ в manifest объявлен пакет `tk.glucodata`

### 3. Дашборд

- **Parent dashboard** (`/api/parent/dashboard/{childId}/...`): KPI за 24 ч, TIR, сравнение периодов, timeline событий, паттерны — 5 эндпоинтов с `CanAccessChildAsync` перед каждым вызовом данных
- **Doctor dashboard** (`/api/doctor/...`): список пациентов с TIR/7 дней и сортировкой, когортная сводка, заметки к измерениям
- **General dashboard** (`/api/dashboard/{childId}/...`): история измерений с фильтрами, сводка
- На вкладке измерений родительского кабинета график располагается над списком измерений, поддерживает день, неделю, месяц и произвольный период; экспорт CSV формируется в UTF-8 с BOM для корректного открытия в Excel
- Карта критического события открывается активной ссылкой; в настройках родителя выбирается провайдер `yandex`, `google` или `openstreetmap`, по умолчанию используется Яндекс Карты
- Уведомления имеют тип, иконку, состояние прочтения и могут помечаться прочитанными по отдельности или все сразу. Сервер ограничивает общее число уведомлений каждого получателя 500 записями и удаляет самые старые независимо от признака прочтения
- Врачебный и административный кабинеты используют тот же адаптивный shell и верхнюю панель, что и родительский кабинет; на мобильной ширине навигация и таблицы перестраиваются для экрана телефона

### 4. Рюкзак (Backpack)

- Снэки с хлебными единицами: `GET/POST /api/backpack/{childId}`, `DELETE /api/backpack/{itemId}`, `POST /api/backpack/{itemId}/consume`
- При удалении и потреблении создаётся запись в `BackpackHistory`
- При потреблении создаётся `SnackConsumptionLog` + Telegram-уведомление родителям
- Авторизация через `IChildAccessService.CanAccessChildAsync` (в контроллере перед вызовом сервиса)
- Ежедневная очистка рюкзака в полночь по часовому поясу ребёнка (`MidnightCleanupJob`)

### 5. Дневник питания и инсулина

- Записи приёмов пищи, перекусов и инъекций: ХЕ, фактически введённый инсулин, время, тип приёма, комментарий
- Для ручного измерения ребёнок выбирает дату и время: по умолчанию подставляется текущее локальное время, ввод времени из будущего ограничен
- Измерение во время перекуса может существовать без самого перекуса; при наличии перекуса оно связывается с записью питания и корректно отображается в дневнике
- Редактируемое расписание доступно в мобильном приложении ребёнка и web-кабинете родителя
- Родитель может вести дневник за ребёнка, если ребёнок маленький или временно без телефона
- Напоминания о еде, перекусах и инсулине настраиваются в мобильном приложении и кабинете родителя
- Статистика отображается как графики и таблицы: ХЕ, инсулин, выполнение расписания, связь с измерениями глюкозы
- Экспорт дневника и статистики в CSV/PDF

### 6. Ачивки и мотивация

- Детские ачивки за регулярные измерения, стабильный сахар, ведение дневника, полезные перекусы и соблюдение расписания
- Ачивки отображаются в мобильном приложении и могут попадать в события родительского кабинета
- Интерфейс поддерживает скины: нейтральный, для мальчика и для девочки; меняются палитра и маскот, структура экранов не меняется

### 7. Приглашения и связи (InviteCodes)

- Генерация кода: `POST /api/invite-codes` — код `N`-символов из алфавита константы `InviteCodeLimits`, TTL 48 ч, дублирующиеся активные коды аннулируются
- Активация кода (`CLAIM`): проверка статуса, TTL, роли пользователя (должна совпадать с `TargetRole`), создание `ParentChildLink` или `DoctorChildLink`
- Управление связями: `GET/DELETE /api/invite-codes/{childId}/links`
- Очистка истёкших кодов через Hangfire (`InviteCodeRetention = 7 дней`)

### 8. Подключение Telegram

- `ConnectionCode` — 8-символьный код вида `ABCD-1234`, хранится как HMAC-SHA256 хеш (`HmacConnectionCodeHasher`, ключ из env `CONNECTION_CODE_KEY`)
- Привязка аккаунта через Telegram Bot: код вводится в боте, устанавливается `TelegramId` у пользователя
- Ежедневная сводка в 21:00 по московскому времени при включённой настройке сохраняется в кабинете родителя и независимо отправляется на email, в Telegram и MAX, если соответствующий канал привязан

### 9. Настройки диабета (DiabetesSettings)

- `TargetRangeMin`, `TargetRangeMax`, `CarbInsulinRatio`, `InsulinSensitivity`
- `GET/PUT /api/diabetes-settings/{childId}` — upsert, создаётся автоматически при создании ребёнка

### 10. Экспорт данных

- CSV-экспорт: `POST /api/export-jobs` → 202 Accepted с `jobId`; фоновая обработка через `ExportJobProcessor` (Hangfire); `GET /api/export-jobs/{id}/download`
- Лимит 100 000 измерений на экспорт
- Очистка файлов и записей через `CleanupJob` (`ExportJobRetention = 30 дней`)

### 11. Sync Log

- Журнал синхронизации мобильного приложения: `GET /api/sync-logs`, фильтр по `childId` и `onlyConflicts`
- Разрешение конфликтов: `PUT /api/sync-logs/{id}/resolve`, `PUT /api/sync-logs/resolve-all`

### 12. Telegram-бот и MAX

- Авторизация через `bot-login` (ServiceAccount JWT)
- Контекст пользователя: `BotUserContext` — текущий выбранный ребёнок, `LastActivityAt`
- Уведомления: измерения, потребление снэков, критические алерты с геолокацией
- Inline-клавиатура для выбора ребёнка из `ParentChildLinks`
- MAX-интеграция вынесена в отдельный проект `SugarGuard.MaxBot`. API использует `IMaxBotClient` и `IMaxBotService`; отправка выполняется только при корректно настроенном токене MAX
- Для MAX реализованы уведомления о новых измерениях, критических событиях и ежедневной сводке. Подключение пользователя и реальный обмен с MAX требуют рабочего токена и проверки на тестовом аккаунте

### 13. AI-рекомендации и родительский контроль

- `IAIRecommendationService` → `GigaChatService` (OAuth2, retry + circuit breaker через Polly)
- `GigaChatTokenCache` — double-check locking на `SemaphoreSlim`, TTL токена 25 мин
- Local fallback: детерминированные рекомендации по статусу глюкозы при недоступности GigaChat
- Хранение в `AIRecommendation`, лимит частоты вызовов через `IAIRecommendationRepository.CountForChildAsync`
- Контекст AI формируется с учётом профиля ребёнка, последних измерений, истории питания и перекусов, параметров диабета и содержимого рюкзака; модель получает явное ограничение не рекомендовать продукты, которых нет в рюкзаке
- Полный аудируемый диалог хранится в `AiConversation` и `AiConversationMessage`: запрос ребёнка, ответ GigaChat и локальный безопасный ответ приложения различаются по источнику
- Родительский раздел «Контроль ИИ» показывает пары запрос–ответ в обратном хронологическом порядке, по пять пар на странице, с указанием источника ответа и статуса проверки безопасности
- Администратор видит агрегированный расход токенов ИИ по детям; это техническая статистика, а не медицинский показатель

### 14. Поддержка, статьи и администрирование

#### Поддержка

- Обращение создаётся в кабинете пользователя с темой, текстом, callback-email, необязательным логом клиента и вложением
- `SupportConversation` и `SupportMessage` формируют диалог пользователя и поддержки; статусы: ожидание ответа поддержки, ожидание ответа пользователя и закрыто
- Новое обращение создаёт уведомление для сотрудников поддержки и отправляет письмо на адрес поддержки; ответ сотрудника создаёт уведомление в кабинете пользователя и отправляется на его callback-email или email учётной записи
- Сотрудник поддержки может отмечать сообщения прочитанными, отвечать и закрывать/повторно открывать обращение. Закрытые обращения визуально выделяются

#### Статьи

- Публикации хранятся в `FaqArticle`; опубликованные статьи доступны родителю, врачу и администратору
- Администратор создаёт, редактирует, публикует и снимает с публикации статьи через административный интерфейс

#### Администрирование

- `GET/PUT /api/admin/users-roles` — список пользователей, смена роли
- `GET/DELETE /api/admin/users-roles/links/parent-child|doctor-child` — управление связями
- Audit log: `GET /api/admin/audit-logs` с фильтрами по `actorUserId`, `action`, `from/to`
- Onboarding funnel: `GET /api/admin/onboarding-funnel`
- Системная статистика: `GET /api/admin/system-stats`
- Управление приглашениями: `GET/POST/DELETE /api/admin/users-roles/invitations`
- Hangfire Dashboard: `/hangfire` — только для авторизованных (⚠️ в продакшне нужна проверка роли `Admin`)

### 15. Health check

- `GET /api/health` — проверка подключения к БД через `IHealthService`

### 16. Расписание измерений (MeasurementSchedule)

- `ScheduleEntry` — CRUD расписания напоминаний, привязан к `Child`

### 17. Ежедневная очистка (CleanupJob)

Hangfire recurring job в 01:00 UTC, удаляет:

| Тип | Критерий | Retention |
|---|---|---|
| `RefreshToken` | истёкшие + отозванные | 30 дней |
| `ConnectionCode` | истёкшие | по `ExpiresAt` |
| `InviteCode` | Expired / Claimed / Rejected | 7 дней |
| `ExportJob` | completed / failed + CSV-файлы | 30 дней |
| `AuditLog` | все старые | 90 дней (GDPR / 152-ФЗ) |
| `SyncLog` | все старые | 30 дней |

## Обновление мобильного приложения

- Мобильное приложение запрашивает `GET /api/mobile-app/version` при запуске
- Endpoint возвращает номер версии, абсолютную ссылку на APK и текст изменений из конфигурации `MobileApp:Android`
- Если доступная версия выше `AppInfo.VersionString`, пользователь видит предложение скачать обновление; по подтверждению открывается ссылка на APK в браузере устройства
- Текущая опубликованная версия — `1.5`, файл распространяется как `SugarGuard.Junior-1.5.apk`. Номер версии включён в имя файла, чтобы пользователь мог однозначно определить скачанный выпуск
- Сайт отдаёт APK с MIME-типом `application/vnd.android.package-archive`; ссылка работает как на десктопных, так и на мобильных браузерах

## Middleware pipeline (порядок)

1. `HttpLogging` — метод, путь, статус, длительность
2. `GlobalExceptionHandlerMiddleware` — перехват необработанных исключений → ProblemDetails
3. `HttpsRedirection`
4. `StaticFiles` — раздача загруженных фото (`/uploads/children/{id}/file`)
5. Security headers inline: `X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy`, `Permissions-Policy`
6. `Cors` (политика `WebClient`)
7. `Authentication` (JWT Bearer)
8. `Authorization`
9. `RateLimiter`
10. `MapControllers`

`AuditLogMiddleware` логирует POST/PUT/PATCH/DELETE (кроме health, metrics, hangfire, swagger).

## Безопасность

### Реализовано

- **JWT + Refresh Token Rotation** — ротация при каждом refresh, массовый отзыв при компрометации
- **AES-256-GCM** для PHI-данных — уникальный nonce на каждое шифрование, GCM tag обеспечивает целостность
- **HMAC-SHA256** для кодов подключения — ключ из env `CONNECTION_CODE_KEY`, минимум 256 бит
- **PBKDF2-SHA256, 600 000 итераций** для паролей — соответствует OWASP 2024
- **`CryptographicOperations.FixedTimeEquals`** — защита от timing-атак при верификации кодов
- **IDOR-защита** — `IChildAccessService.CanAccessChildAsync` с L1-кешем (`HttpContext.Items`) и L2-кешем (`IMemoryCache`, TTL 10 мин)
- **Rate Limiting** — отдельные лимиты для `/auth/login` (5 req/min/IP) и `/auth/bot-login`
- **Security headers** — `X-Content-Type-Options`, `X-Frame-Options: DENY`, `Referrer-Policy`
- **CORS** — whitelist по конфигурации `AllowedOrigins`
- **Аудит** — все мутирующие операции логируются в `AuditLog` через middleware и сервисы
- **CGM-данные** — поток Juggluco проходит синтаксическую валидацию и сохраняется локально до сетевой синхронизации; серийный номер сенсора не записывается в базу SugarGuard

### Известные ограничения (требуют решения перед продакшном)

| # | Проблема | Где | Приоритет |
|---|---|---|---|
| 1 | `HangfireAuthorizationFilter` проверяет только `IsAuthenticated`, не роль `Admin` | `HangfireAuthorizationFilter.cs` | 🔴 Критический |
| 2 | `BackpackService.ConsumeAsync` — нет `CanAccessChildAsync` на уровне сервиса (в контроллере есть) | `BackpackService.cs` | 🟠 Средний |
| 3 | Shared cache key в `IsDoctorForChildAsync` и `IsParentOfChildAsync` — оба используют `CanAccessCacheKeyPrefix`, могут возвращать некорректный результат в одном запросе | `ChildAccessService.cs` | 🟠 Средний |
| 4 | `DoctorChildLinks` без `IsActive` в `ChildrenService.GetAccessibleAsync` — деактивированный врач видит детей в списке | `ChildrenService.cs` | 🟠 Средний |
| 5 | `SnackConsumptionLog.DeletedBy` содержит строку-заглушку `"consumed:userId:guid"` вместо реального ID | `BackpackService.cs` | 🟡 Низкий |
| 6 | `IMemoryCache` в `ChildAccessService` и `VerificationService` — single-instance only, не подходит для горизонтального масштабирования | Оба сервиса | 🟡 Низкий (при масштабировании) |

## Нереализованный функционал (backlog)

- OAuth 2.0 (Яндекс): `WebAuthenticator` в MAUI, эндпоинт `GET /api/auth/yandex` на бэкенде
- SMS-верификация (SMSC.ru / SMS.ru)
- MFA / TOTP для Admin и SupportAdmin
- Key Rotation для PHI AES-ключей
- Concurrency Token (ETag / `RowVersion`) на `Measurement`, `BackpackItem`, `DiabetesSettings`
- OpenTelemetry + correlation-id (Serilog)
- API Versioning (`/api/v1/...`)
- Web Push PWA manifest + service worker для Blazor
- AndroidKeyStore явная настройка для `SecureStorage`
- `IDistributedCache` (Redis) вместо `IMemoryCache` для multi-instance деплоя
- Подтверждение работы Libre/Juggluco на физических Android-устройствах и матрица совместимости конкретных моделей сенсоров
- Прямая интеграция с официальными API производителей сенсоров — только после отдельной юридической и технической оценки
